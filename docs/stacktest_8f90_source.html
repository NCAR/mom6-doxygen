20! call a subroutine that has large local variables on the stack
21! to make sure the stacksize limits aren't too small. in this
22! little test it's easy to see what's going on, but when you run
23! filter it tends to die in random places if you've got stack
24! limit problems and isn't easy to diagnose.

41subroutine orange_moose(count)
42 integer*8, intent(in) :: count
43
44! large local variable:
45integer :: godzilla(count)
46character(32) :: junkbuf
47integer :: bob
48
49! access the array contents to be sure we have enough stack space.
50! in some of my tests the program did not crash on either of the
51! varieties of assignment statement (disturbing), but then did
52! crash with a stack exception when the values were accessed in
53! the write statement. makes you wonder how much of memory was
54! assigned 123, and whether it over-wrote some random memory location.
55
56! should initialize the entire array, but doesn't always trigger a crash
57! even if this is too large to fit on the stack.
58godzilla(:) = 123
59
60! try explicitly accessing the first and last elements. again, not always
61! the fatal lines.
62godzilla(1) = 456
63godzilla(count) = 789
64
65! this seemed to really make it die - not sure why this is different
66! than the assignments above -- and it's an integer array so it's not
67! a random illegal bit pattern causing a floating point trap - any bit
68! string is a valid integer value.
69write(junkbuf, "(2I4)") godzilla(1), godzilla(count)
70
71! now take the local var and explicitly pass it as an argument.
72! some compilers can be told to put large local variables on the heap.
73! but i believe that subroutine arguments have to be on the stack.
74bob = deeper(godzilla)
75write(junkbuf, "(I4)") bob
76
77! if you get here, it is as good a test as i can devise.
78! seems like the stack size is going to be ok.
79
80end subroutine
