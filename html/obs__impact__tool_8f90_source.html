5!> This program users to construct a table that is read by filter at
6!> run-time to localize the impact of sets of observation types on sets of
7!> state vector quantities.
8
9!> The standard DART algorithms compute increments for an observation and then
10!> compute corresponding increments for each model state variable due to that
11!> observation. To do this, DART computes a sample regression coefficient using
12!> the prior ensemble distributions of a state variable and the observation. The
13!> increments for each member of the observation are multiplied by this
14!> regression coefficient and then added to the corresponding prior ensemble
15!> member for the state variable. However, in many cases, it is appropriate to
16!> reduce the impact of an observation on a state variable; this is called
17!> localization. The standard DART algorithms allow users to specify a
18!> localization that is a function of the horizontal (and optionally vertical)
19!> distance between the observation and the state variable. The localization is
20!> a value between 0 and 1 and multiplies the regression coefficient when
21!> updating state ensemble members.
22!>
23!> Sometimes, it may be desirable to do an additional localization that is a
24!> function of the type of observation and the state vector quantity. This
25!> program allows users to construct a table that is read by filter at run-time
26!> to localize the impact of sets of observation types on sets of state
27!> vectorquantities. Users can create named sets of observation types and sets
28!> of state vector quantities and specify a localization for the impact of the
29!> specified observation types on the state vector quantities.
30!>
31!> An example would be to create a subset of observations of tracer
32!> concentration for a variety of tracers, and a subset of dynamic state
33!> variable quantities like temperatures and wind components. It has been common
34!> to set this localization value to 0 so that tracer observations have no
35!> impact on dynamic state quantities, however, the tool allows values between 0
36!> and 1 to be specified.
37!>
38!> All the listed observation types and state vector quantities
39!> must be known by the system. If they are not, look at the
40!> &preprocess_nml :: input_items namelist which specifies
41!> which obs_def_xxx_mod.f90 files are included, which is
42!> where observation types are defined. Quantities are defined
43!> in the assimilation_code/modules/observations/DEFAULT_obs_kinds_mod.F90 file.
44!> (Note you must add new quantities in 2 places
45!> if you do alter this file.)
